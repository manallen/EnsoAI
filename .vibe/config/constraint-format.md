# 约束格式规范

定义 vibe 工作流中约束集的标记语法和输出格式。

## 约束强度标记

### 三级强度

| 强度 | 标记 | 含义 | 处理 |
|------|------|------|------|
| MUST | 必须 | 硬性约束，不可违反 | 技术方案必须满足 |
| SHOULD | 应该 | 强建议，除非有充分理由 | 技术方案优先考虑 |
| MAY | 可以 | 可选，视情况采纳 | 技术方案参考 |

### 语法格式

```markdown
- [HC-1] **MUST** 保持现有 ASCII 图表可视化风格 — 来源: 用户
- [SC-1] **SHOULD** 约束强度使用简化三级 — 来源: Adversary 审查
- [SC-4] **MAY** Given/When/Then 作为可选扩展 — 来源: Adversary 审查
```

## 约束表格格式

### 硬约束表格 (MUST)

```markdown
### 硬约束 (MUST)

| ID | 约束描述 | 来源 | 域 |
|----|----------|------|-----|
| [HC-1] | 必须保持现有 ASCII 图表可视化风格 | 用户 | UI |
| [HC-2] | 必须支持 UI/后端/数据三类项目的需求探索 | 用户 | 系统 |
| [HC-3] | vibe:explore 的需求对齐是最高优先级 | 用户 | 业务 |
```

### 软约束表格 (SHOULD / MAY)

```markdown
### 软约束 (SHOULD / MAY)

| ID | 约束描述 | 强度 | 来源 | 优先级 |
|----|----------|------|------|--------|
| [SC-1] | 约束强度使用简化三级 | SHOULD | Adversary | 高 |
| [SC-2] | 根据项目规模自适应门控数量 | SHOULD | Adversary | 中 |
| [SC-4] | GWT 作为可选扩展 | MAY | Adversary | 低 |
```

### 优先级定义

| 优先级 | 权重 | 说明 |
|--------|------|------|
| 高 | 1.0 | 优先实现，影响核心功能 |
| 中 | 0.6 | 建议实现，影响用户体验 |
| 低 | 0.3 | 可延后，属于优化项 |

## 域偏重显示格式

### 进度条语法

使用 ASCII 圆点表示域偏重程度：

```
●●●●● = 100% (5/5) — 主域
●●●●○ =  80% (4/5)
●●●○○ =  60% (3/5) — 次域
●●○○○ =  40% (2/5)
●○○○○ =  20% (1/5)
○○○○○ =   0% (0/5) — 无关
```

### 域偏重行格式

```markdown
**域偏重**: [UI ●●●●●] [后端 ●●○○○] [数据 ○○○○○]
```

```
  数据约束占比:    10% → ●○○○○
```

## GWT 可选扩展格式

### 基本语法

使用 Markdown 缩进格式，而非 HTML `<details>` 标签：

```markdown
### 成功判据

| ID | 描述 | 验证方式 |
|----|------|----------|
| [OK-1] | 用户在需求探索阶段能够明确表达意图 | E2E 测试 |

**[OK-1] 详细场景** (可选展开):

    Scenario: 成功完成需求探索
      Given 用户启动 /vibe:explore
      And 用户提供项目描述 "实现一个 Todo 应用"
      When 系统完成域类型检测
      Then 显示域偏重确认框
      And 用户确认后进入问题队列

    Scenario: 单问题交互正常
      Given 用户处于问题确认阶段
      When 系统显示问题 "目标用户是谁？"
      Then 用户可以选择预设选项 [A/B/C]
      Or 用户可以输入自定义回答
      And 系统记录回答后显示下一问题
```

### 何时使用 GWT

- 成功判据需要精确行为定义时
- 涉及多个步骤的用户流程
- 需要区分正常流程和异常流程

### 何时不使用 GWT

- 简单的功能验证 (如 "API 返回 200")
- 非用户交互的技术约束 (如 "响应时间 < 200ms")
- 约束本身已经足够明确

## 完整示例

### requirements.md 约束集部分

```markdown
## 约束集

### 域偏重

**域偏重**: [UI ●○○○○] [后端 ●●●●○] [数据 ●●○○○]

### 硬约束 (MUST)

| ID | 约束描述 | 来源 | 域 |
|----|----------|------|-----|
| [HC-1] | 必须保持现有 ASCII 图表可视化风格 | 用户 | UI |
| [HC-2] | 必须支持 UI/后端/数据三类项目 | 用户 | 系统 |
| [HC-3] | vibe:explore 需求对齐是最高优先级 | 用户 | 业务 |
| [HC-4] | 单问题原则与并行探索平衡 | Adversary | 系统 |
| [HC-5] | 递进式深挖有明确退出条件 | Adversary | 业务 |

### 软约束 (SHOULD / MAY)

| ID | 约束描述 | 强度 | 来源 | 优先级 |
|----|----------|------|------|--------|
| [SC-1] | 约束强度使用简化三级 | SHOULD | Adversary | 高 |
| [SC-2] | 根据项目规模自适应门控 | SHOULD | Adversary | 中 |
| [SC-3] | CONTEXT 合并到 STATE.md | SHOULD | Adversary | 高 |
| [SC-4] | GWT 作为可选扩展 | MAY | Adversary | 低 |
| [SC-5] | 灰色地带按域类型自适应 | SHOULD | GSD | 中 |

### 依赖关系

```
[vibe:explore 优化] ──requires──▶ [vibe:tech/plan/exec 更新]
         │
[STATE.md 扩展] ──affects──▶ [所有 phase]
```

| ID | From | To | 类型 | 说明 |
|----|------|-----|------|------|
| [DEP-1] | vibe:explore | vibe:tech/plan/exec | requires | 需求文档格式变化 |
| [DEP-2] | STATE.md | 所有 phase | affects | 状态管理变化 |

### 风险

| ID | 风险描述 | 影响 | 缓解策略 |
|----|----------|------|----------|
| [RISK-1] | 门控过多导致用户疲劳 | 中 | 提供快速模式 |
| [RISK-2] | 灰色地带识别不准确 | 中 | 域类型检测 + 用户确认 |
| [RISK-3] | 退出条件过早 | 中 | 保留"继续深挖"选项 |

### 成功判据

| ID | 描述 | 验证方式 |
|----|------|----------|
| [OK-1] | 用户能够明确表达意图 | E2E 测试 |
| [OK-2] | 约束集可直接指导下游 | 技术选型验证 |
| [OK-3] | 不同类型项目有差异化问题 | 多场景测试 |
| [OK-4] | 递进式深挖能收敛 | 轮次统计 |
| [OK-5] | 现有用户可平滑迁移 | 兼容性测试 |

**[OK-1] 详细场景** (可选展开):

    Scenario: 用户明确表达意图
      Given 用户启动 /vibe:explore
      When 用户回答完所有必问问题
      Then 系统生成的需求摘要与用户预期一致
      And 用户确认 "理解正确"
```

## 向后兼容

### 现有格式支持

旧格式（无强度标记）仍然有效，系统自动推断：

```markdown
# 旧格式
- [HC-1] 必须支持... — 来源：用户

# 自动转换为
| ID | 约束描述 | 来源 | 域 |
|----|----------|------|-----|
| [HC-1] | 必须支持... | 用户 | (自动推断) |
```

### 迁移建议

- 新项目使用新格式
- 旧项目无需迁移，新格式向后兼容
- 如需迁移，在 vibe:explore 阶段自动升级
